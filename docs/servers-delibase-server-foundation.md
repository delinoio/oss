# servers-delibase-server-foundation

## Scope
- Project/component: `delibase` / `server`
- Canonical path: `servers/delibase`
- Canonical future API origin: `https://delibase.deli.dev`

## Runtime and Language
- Runtime: Go service
- Persistence: PostgreSQL with ordered migrations and sqlc for all queries
- API: `delibase.v1` Connect RPC, with signed Polar webhook REST as the explicit non-Connect exception.
- Stateful structure must provide explicit service, contracts, logging, database, generation, and configuration boundaries before runtime rollout.
- The runnable foundation lives at `servers/delibase/cmd/delibase`, registers all six generated handlers, exposes `/healthz` and PostgreSQL-backed `/readyz`, and shuts down gracefully. `AccountService` and the core non-invitation `OrganizationService` RPCs use authenticated PostgreSQL/sqlc transactions. Invitation RPCs and `TeamService`, `BillingService`, and `UsageService` business RPCs remain typed Connect `Unimplemented` responses rather than placeholder successes.

## Users and Operators
- DeliDev users and organization Owners, Admins, Members, Team Admins, and Team Members.
- Authenticated mini-app backend services reporting integer metered usage.
- Maintainers operating PostgreSQL, Logto, Polar, artifact builds, and future GHCR releases.

## Interfaces and Contracts
- Six versioned Connect services: `AccountService` (account state/onboarding/deletion), `OrganizationService` (organizations, slugs, roles, invitations), `TeamService` (hierarchy, moves, subtree deletion, memberships/access), `CatalogService` (anonymous public enabled app/meter metadata and effective-dated USD micro-unit prices with version metadata), `BillingService` (summary, checkout, portal, limits, ledger/usage reads), and `UsageService` (reserve/commit/release).
- Server implementations import generated messages from `github.com/delinoio/oss/protos/delibase/gen/go/delibase/v1` and Connect interfaces from its `delibasev1connect` child package; handlers must not redefine transport contracts.
- Human calls require a Logto user access token except for anonymous `CatalogService` reads. Usage mutations require a Logto M2M token plus the `x-delibase-forwarded-user-token` Connect metadata key carrying a forwarded end-user Logto token. Raw Logto client secrets are provider-side credentials and are never accepted in inbound Connect metadata. Validate issuer, the canonical `https://delibase.deli.dev` audience, expiry, scopes, M2M service meter allowlists, membership, and effective team access server-side; redact that metadata value from all logs, traces, errors, and diagnostics.
- Every HTTP path and Connect procedure must declare an explicit shared authentication mode. Anonymous catalog reads use `ModePublic`; protected route omissions retain the invalid zero value and fail closed.
- Shared authentication attaches only typed Logto user/service claims to context and removes both credential headers before handlers execute. `sub == client_id` identifies the Logto client-credentials/M2M shape; user and M2M validators reject each other's token type. Delibase then resolves the user/service against its own local state and remains the sole authority for organizations, memberships, roles, teams, meter allowlists, billing, and usage decisions.
- List APIs use opaque cursors. `CatalogService` lists only enabled apps/meters using UUID-keyset cursors (default 25, capped at 100), and detail reads safely return `NotFound` for disabled, absent, or not-currently-priced resources. Catalog detail/list meter responses include the exact effective price-version UUID so later usage reservations can pin it. Errors use Connect status codes and vetted stable enum-based reasons for auth, authorization, slug, invitation, team depth/cycle, subscription, overage, reservation, idempotency, deletion, and resource-state failures. Shared authentication distinguishes bearer and forwarded-user reasons; explicit signing-key provider outages fail closed as retryable `Unavailable` dependency failures, while unknown keys and algorithm/type/curve mismatches are invalid credentials. The shared safe-error boundary drops arbitrary response metadata, unrecognized protobuf details, and free-form detail fields; the outer request metadata interceptor adds only validated request and trace IDs to unary and streaming failures.
- Persisted IDs are UUID v7. Local accounts keep their original Logto `sub` for their lifetime. Slugs are globally unique/changeable, and retained aliases cannot be rewritten or deleted outside their organization-deletion cascade. Teams have one organization, nullable parent, names unique among siblings (including root teams), maximum five levels below the organization, downward inherited access, and an undeletable/non-renamable protected `General` team. Hierarchy validation serializes moves within an organization so concurrent updates cannot create a cycle or exceed the depth limit. Deleting another team requires explicit confirmation, deletes its subtree only when no active usage reservation exists there, and historical usage/audit records retain immutable team ID/name snapshots.
- Invitations are hashed bearer links, seven-day valid, revocable, and reusable by distinct users. Preview and acceptance require an authenticated user plus a valid invitation bearer token, but not pre-existing organization membership or team access. The storage boundary locks the invitation, rejects revoked or expired acceptance, and supplies the acceptance timestamp. Acceptance and revocation use distinct idempotency operations; invitation creation is not idempotent. An invitation may grant only Organization Admin or Member; Member invitations must target a team and assign Team Admin or Member. Owner may be granted only by an existing Owner after the user has joined, and existing membership roles are not silently changed.
- Organization roles: Owner (full access and deletion), Admin (management, billing, limits, full usage, but no Owner management or organization deletion), Member (app use, shared credits, and accessible team usage, but no invoices/full ledger/subscription/overage controls). Owners and Admins are implicit Team Admins for every team. Every committed organization has at least one active Owner; organization creation and its initial active Owner membership commit atomically, and account status changes cannot leave an organization without a usable Owner.
- Held usage reservations retain their live organization membership and block organization-member removal or self-leave with `ERROR_REASON_MEMBER_HAS_ACTIVE_RESERVATIONS`.
- Account deletion blocks on last-Owner status, disables local access, removes operational data, queues Logto deletion, signs out, retries external deletion, and retains only pseudonymized financial/audit references for seven years.
- Completed account and organization deletion leaves append-only pseudonymous tombstones with an explicit minimum seven-year retention boundary. A deleted Logto subject is retained only as a SHA-256 digest so an unexpired access token cannot recreate the account after the Management API deletion succeeds. The raw subject exists only in the disabled operational account row while the durable Logto deletion job still needs it and is removed after successful delivery.

## Storage
- PostgreSQL tables/migrations cover users, organizations, memberships, slug aliases, teams, invitations, catalog/meter enabled state, required reservation TTLs, effective-dated USD micro-unit prices, service-identity and integer-unit mappings, and Polar meter mappings, plus Polar subscriptions/periods, append-only ledger entries, reservations, webhook inbox, integration outbox, deletion jobs, and audit events. Each committed organization has exactly one Polar team customer, keyed by the organization UUID as Polar `external_id`, and cannot drop that customer before its organization-deletion cascade. Organization/team-membership identity, Polar subscription identifiers, and Logto service-client identifiers are immutable. An organization has at most one active subscription while canceled, revoked, and otherwise inactive subscription rows remain as immutable provider history.
- Database constraints prevent overlapping price windows for one meter and overlapping billing periods for one organization, keep subscriptions and ledger periods in the billed organization, require usage-linked settlement to identify the billing period containing its storage-derived commit time, snapshot that period window into retained ledger history, validate each ledger entry's running balance under an organization lock, bind reservation price snapshots to the effective catalog version, derive active reservation creation times in storage, bind expiry to the meter TTL, require overflow-safe reservation unit-price totals, reject direct committed inserts and all reservation deletes, require matching credit/overage release entries before release or expiry, revalidate authoritative credit and active-subscription current-period overage capacity at commit, and cap committed units at the reserved maximum.
- Money is signed 64-bit USD micro-units (`10,000,000` = $10.00); usage is signed 64-bit integer units. Reject overflow, negative reservations, and invalid catalog precision.
- Polar supplies one monthly $10 product: each successfully paid cycle grants exactly $10.00 rollover credit; unused credits do not expire; metered overage is invoiced by Polar. Taxes/fees do not alter the grant.
- Delibase ledger is authoritative for operational balance, holds, authorization, and DeliDev display; Polar is authoritative for payment settlement and invoices. Refund/chargeback reverses its grant and makes consumed shortfall overage subject to the limit.
- Owner/Admin must set a non-negative monthly overage limit; zero is default. Committed and held overage count against the current Polar period. Lowering the limit below current committed or held overage blocks new reservations and pending commits that no longer fit authoritative capacity; it never reverses already committed usage or mutates append-only ledger entries. Cancellation/revocation/past-due preserves existing credits but grants no credits and prevents new or pending overage settlement.
- `ReserveUsage` requires organization/team/meter IDs, max units, client reference, and idempotency key; under an organization lock it atomically verifies that deletion has not started, the active organization membership, enabled service/meter mapping, current team-name snapshot, effective catalog price, meter TTL, available settled credit minus active holds, and committed-plus-held overage against an active subscription's effective billing-period limit before accepting the exact credit-first hold split. Reservation identity, snapshots, limits, holds, references, and timestamps are immutable after creation; only a storage-timestamped held-to-committed/released/expired transition is allowed, and committed state can originate only from the validated immutable usage-record insert path. Commit rechecks that the reservation account is active and retains effective team access, that authoritative settled credit still backs active credit holds, and that an active current subscription period still backs committed-plus-held overage; it applies actual units (never above reserved), consumes no more held credit or overage and uses credit first, derives the commit timestamp in storage, releases unused holds, attributes actor/team/service, and queues Polar usage. Release/expiry requires ledger entries returning the full held credit and overage amounts; late commit fails.
- Idempotency is authenticated-caller/operation scoped: human RPCs use the user subject and M2M RPCs use the service identity. The same key and payload returns the original result; a different payload conflicts. PostgreSQL transactions/row locks prevent over-allocation. Polar webhooks are signature-verified and persisted idempotently with a validated JSON-object payload of at most 1 MiB before success. The reusable `internal/reliability` enqueue functions accept transaction-bound sqlc queries so business mutation and queue insertion commit or roll back together. Webhook provider/event identity, integration/operation/idempotency identity, and deletion type/idempotency identity are duplicate-safe; a duplicate with changed immutable input conflicts.
- Reliability handlers are registered only under stable typed handler identifiers derived from allowlisted webhook event, integration operation, or deletion-job values; database or request text never selects an arbitrary handler. Workers claim one due row atomically with `FOR UPDATE SKIP LOCKED`, a unique claim token, and a bounded lease. Completion and failure compare that token, so stale or crashed workers cannot repeat a state transition. Claiming increments the attempt count, including attempts that crash. Expired claims become eligible again; if the twelfth normal attempt crashes, lease recovery transitions it to retained dead-letter state.
- Normal delivery has exactly 12 attempts with capped exponential backoff and injected jitter. A failed twelfth attempt becomes dead letter immediately. Dead-letter rows retain their original event, first dead-letter timestamp, normal-attempt count, and cumulative dead-letter-attempt count; they become eligible every 24 hours without jitter until a handler succeeds. Successful dead-letter delivery keeps retained dead-letter history while setting the terminal processed, delivered, or completed state. Webhook inbox events, integration-outbox events, and deletion jobs cannot be deleted, and retry updates cannot rewrite their event identity/payload, integration-outbox event bodies, deletion-job targets, idempotency key, or pseudonymous actor.

## Security
- Logto is the identity trust boundary; delibase owns local authorization and data. Polar owns hosted payment consent, cards, receipts, invoices, cancellation, and payment recovery; delibase never handles card data.
- Store only invitation token hashes. Never persist or log Logto/Polar tokens, webhook secrets, authorization headers, `x-delibase-forwarded-user-token`, forwarded user tokens, card data, or raw billing PII. Queue payloads reject sensitive key and value shapes before persistence. Append-only audit rows use stable typed event identifiers and accept only empty/system or keyed pseudonymous actor references plus bounded allowlisted request correlation/method/procedure metadata whose correlation identifiers also pass the shared credential-shape redaction boundary. Decision and result are stored on every row and no audit update or delete path exists.
- Routine actor fields must use only the exact keyed `actor:v1:<32-lowercase-hex>` output, and the root redacting handler must recursively sanitize typed diagnostic maps and slices.
- PostgreSQL failure fails mutations closed. During Polar outage, authorize within local credits/overage and queue events; checkout creation makes no local mutation if Polar is unavailable.
- Organization deletion is one-way once `deleted_at` is set: it hides access immediately, blocks usage, forfeits remaining credits without refund, queues Polar cancellation, deletes operational org/team/member data, and retains seven-year pseudonymized financial/audit records. Active reservations retain live organization/team/membership references and therefore block deletion; finalized reservation and usage rows retain immutable identifier/name snapshots without keeping operational rows alive.

## Logging
- Use `log/slog` structured events with request/trace ID, pseudonymous actor, organization/team/service/meter/reservation IDs, decision, result, and safe error classification.
- Audit authorization, organization/role/invitation/team, billing-limit, checkout/subscription/refund, reservation/settlement, deletion, and webhook decisions immutably without secrets or raw PII.

## Build and Test
- Shared-contract checks are `pnpm check:proto`, `go test ./protos/delibase/...`, and `go vet ./protos/delibase/...`. Once server implementation exists, also run `gofmt -w`/format check, `go vet ./...`, `go test ./servers/delibase/...`, sqlc generation/checks, PostgreSQL integration/concurrency tests, migration checks, and Docker build validation. The Docker check must validate a minimal multi-stage image that runs as non-root and exposes working health/readiness behavior.
- Current server checks include `servers/delibase/scripts/generate-sqlc.sh` followed by a clean generated diff, `servers/delibase/scripts/test-postgres.sh` for ephemeral PostgreSQL migration/idempotency and rollback validation, and `servers/delibase/scripts/test-image.sh` for a minimal non-root image with live health/readiness validation.
- Reliability clocks, jitter sources, and claim-token generators are injectable so retry scheduling, lease recovery, and claim ownership are deterministic in tests. PostgreSQL coverage includes duplicate enqueue, concurrent claims, crash recovery, retry timing, the twelfth-attempt transition, daily dead-letter recovery, claim-token compare-and-set transitions, immutable audit rows, and credential rejection.
- Account/organization service PostgreSQL coverage races replayed organization creation, verifies the atomic Owner/`General`/direct-Team-Admin bundle, exercises slug aliases and canonical resolution, multiple-Owner and Admin restrictions, last-Owner blockers, immediate access removal, Logto deletion retries, deleted-subject tombstones, and transactional organization deletion.
- CI runs the same sqlc reproducibility check and delibase test suite against an ephemeral PostgreSQL 17 service whenever the delibase server, generated Go API, or shared server infrastructure changes.
- CI must test duplicate/reordered webhooks, idempotency, concurrent reservations, account/organization/team rules, five-level/cycle constraints, invitation role boundaries, protected-team deletion, catalog validation with fixtures covering at least two apps and meters with required reservation TTLs, billing state, outages, and deletion retention behavior.
- Release CI must add `release-delibase.yml`, trigger only on `delibase@v*` tag pushes, and publish only signed `ghcr.io/delinoio/delibase:vX.Y.Z` and `:latest` multi-architecture (`linux/amd64`, `linux/arm64`) images with SPDX SBOM, provenance, and health/readiness plus non-root validation; no `edge`, SHA, or main-branch images.
- This documentation change does not activate a service, deploy an API, or publish an image; the release workflow and artifacts are issue #722 implementation deliverables.

## Dependencies and Integrations
- Consumes root sources at `protos/delibase/v1` and shared packages under `servers/internal`.
- Integrates with DeliDev at `https://deli.dev`, Logto, Polar, PostgreSQL, and future GHCR artifact publication.
- Catalog is checked-in validated configuration with explicit mini-app metadata, app/service UUID v7 identities, app/meter enabled state, required reservation TTLs, effective-dated signed-int64 USD micro-unit prices, canonical precision-zero integer unit definitions, service-to-meter allowlists, and Polar meter mappings. The production file may intentionally be empty; test fixtures contain at least two apps and meters. Synchronization runs transactionally and idempotently after migrations/startup, disables omitted mutable entries, retains referenced mappings, and inserts/ends price versions instead of rewriting price history. Startup validation rejects malformed documents, duplicate IDs, invalid/negative TTLs or prices, invalid precision, overflow-invalid values, missing or duplicate unit/service/Polar mappings, invalid mapping references, and incompatible price dates; disabled entries are excluded from anonymous reads and new usage authorization, while active reservations retain their service allowlist and exact Polar meter mapping until finalized, and effective-dated price changes pin reservation versions. No runtime catalog mutation API or operator UI.
- Configuration ownership: this service owns server non-secret configuration, database/catalog settings, CORS, API origin, and secret environment variable names; Logto/Polar own provider-side configuration; DeliDev owns browser-safe client configuration; shared defaults belong to `servers/internal`. The canonical server variables are `DELIBASE_API_ORIGIN`, `DELIBASE_CORS_ALLOWED_ORIGINS`, `DELIBASE_CATALOG_PATH`, `DELIBASE_LOGTO_ISSUER`, `DELIBASE_LOGTO_AUDIENCE` (which must equal `https://delibase.deli.dev`), and `DELIBASE_LOGTO_JWKS_URL` (an HTTPS URL without credentials, query, or fragment), plus `DELIBASE_DATABASE_URL`, `DELIBASE_LOGTO_M2M_CLIENT_ID`, `DELIBASE_LOGTO_M2M_CLIENT_SECRET`, `DELIBASE_POLAR_ACCESS_TOKEN`, `DELIBASE_POLAR_WEBHOOK_SECRET`, and the distinct minimum-32-byte `DELIBASE_LOG_PSEUDONYM_KEY` (secret). The browser-safe client variables are `PUBLIC_DELIBASE_API_ORIGIN`, `PUBLIC_LOGTO_ENDPOINT`, `PUBLIC_LOGTO_APP_ID`, and `PUBLIC_LOGTO_AUDIENCE`; they must not contain secrets and the audience must use the same canonical value. No variable may be logged, and required variables fail closed when absent. The implementation must add the matching CI/deployment templates when the runtime is introduced.
